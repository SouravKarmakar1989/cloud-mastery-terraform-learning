### 1. Encryption at Rest: Four Models, One Decision

S3 offers four server-side encryption models. Choosing wrong means either leaving data unprotected, losing the ability to audit key usage, or creating operational complexity that breaks applications. The decision should be made at the bucket level with a default encryption setting, enforced by bucket policy, and never left as an application-level opt-in.

**SSE-S3 (Server-Side Encryption with S3-Managed Keys).** AWS manages the encryption master key entirely. Every object is encrypted with AES-256 using a unique key, and that key is itself encrypted with a rotating master key that AWS manages and stores. There is no per-request cost beyond the base API request charge. No CloudTrail event is generated for key usage. The consequence of this simplicity: you have no way to audit who decrypted what, no way to revoke access at the key level, and no way to bring your own key material. For public datasets, logs, or non-sensitive content — this is appropriate. For anything with compliance requirements, it is insufficient.

**SSE-KMS (Server-Side Encryption with AWS KMS Keys).** Each object is encrypted with a data key generated by KMS, and that data key is encrypted by the KMS key you specify. Every PUT and GET operation that requires key material generates a `GenerateDataKey` or `Decrypt` call to KMS, which is logged to CloudTrail. This full audit trail — who decrypted which object, from which IAM identity, at what timestamp — is the core value of SSE-KMS. Downsides: per-request KMS API costs (approximately $0.03 per 10,000 requests), and KMS request rate limits (burst of 30,000 requests/second in most regions, shared across all services using KMS in the account/region). For high-throughput workloads executing millions of GET requests per second, the KMS request rate can become a bottleneck. **Bucket Key** addresses this: enable the S3 Bucket Key to cache the data encryption key at the bucket level, reducing KMS API calls by up to 99% and eliminating the rate limit concern for most workloads.

**SSE-C (Server-Side Encryption with Customer-Provided Keys).** The client provides the encryption key on every request. S3 uses the key for the operation and immediately discards it — AWS never stores the customer-provided key. The client must keep a catalog of which key was used for which object, because AWS cannot decrypt the object on your behalf. This model is appropriate for organizations with strict requirements to retain all key material on-premises. Operationally, it is complex: lost key = permanently inaccessible object.

**CSE (Client-Side Encryption).** The client encrypts the data before sending it to S3. AWS stores opaque ciphertext — the platform has no role in encryption or decryption. The client is fully responsible for key management. The AWS Encryption SDK and language-specific SDKs provide client-side encryption support. This is the correct model for organizations whose compliance requirements explicitly require encryption before data leaves their network perimeter.

**Enforcing Encryption via Bucket Policy.** Default encryption ensures that objects written without an encryption header are encrypted with the bucket default. But a well-meaning application might explicitly specify an unencrypted PUT header and override the default. To prevent this, add a bucket policy that explicitly denies PUT requests that do not carry the expected server-side encryption header:
```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::your-bucket/*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-server-side-encryption": "aws:kms"
    }
  }
}
```
This denies any PUT that does not specify SSE-KMS, regardless of the bucket default.

### 2. Encryption in Transit: TLS Is Not Optional

Encryption at rest protects data in the storage medium. Encryption in transit protects data on the network between the client and S3. S3 supports both HTTP and HTTPS endpoints — you must enforce HTTPS through a bucket policy because S3 will accept HTTP requests by default.

**Why in-transit encryption matters for storage.** A man-in-the-middle attacker on the same network segment could intercept unencrypted S3 API traffic and read object contents, capture authentication credentials (SigV4 signatures are time-limited but the window exists), or inject modified object data. Enforcing HTTPS-only via the `aws:SecureTransport` condition in the bucket policy closes this attack surface entirely.

**TLS certificate validation.** AWS S3 endpoints serve valid TLS certificates for the `s3.amazonaws.com` and regional endpoint domains. Applications should always validate TLS certificates — passing `ssl_verify=False` in an S3 client call is not a shortcut for learning; it disables the primary protection against MITM attacks.

**VPC endpoint and data in transit.** Traffic from EC2 instances in a VPC to S3 through a Gateway VPC Endpoint travels through the AWS backbone and never traverses the public internet. It still passes through the NVMe-and-network fabric of the AWS network, which is encrypted by the underlying hardware in Nitro-based regions. Gateway endpoints do not use TLS at the application layer separately from the HTTPS S3 request itself — HTTPS enforcement still matters.

### 3. S3 Object Lock: WORM Storage for Compliance and Governance

Object Lock places a time-limited Write Once Read Many (WORM) retention hold on S3 objects. A locked object cannot be overwritten or deleted for the retention duration, regardless of which IAM principal attempts the delete. This is the mechanism for meeting SEC 17a-4(f), CFTC 1.31, FINRA, HIPAA, and other regulatory requirements that mandate immutable audit records.

**Object Lock must be enabled at bucket creation.** You cannot enable Object Lock on an existing bucket. Once enabled, versioning is also automatically enabled (Object Lock requires versioning). A versioned, Object Lock-enabled bucket is the archival data store pattern for regulatory workloads.

**Retention modes: Compliance vs Governance.**

- **Compliance mode.** No one — including the root user — can delete the object or shorten the retention period before expiry. Not even AWS can override this. This is the strongest immutability guarantee. Use for financial records, medical data, and any data that a regulator will audit. The trade-off: if you configure the wrong retention period, you cannot fix it.

- **Governance mode.** IAM principals with the `s3:BypassGovernanceRetention` permission can override the lock and delete or overwrite the object. Provides protection from accidental deletion while still allowing authorized personnel to rectify mistakes. Use for internal operational data retention where you want protection but not regulatory-grade immutability.

**Legal Holds.** A legal hold places an indefinite WORM lock on an object with no expiry date. It remains in place until explicitly removed by a principal with `s3:PutObjectLegalHold` permission. Legal holds are used during litigation discovery: place a legal hold on all relevant objects, maintain it until the case closes, then remove it.

**Default retention.** A bucket can have a default retention period. Any object written to the bucket without an explicit retention setting inherits the bucket default. This is important for automated data ingestion pipelines — configure the default retention at the bucket level, and every object written by the pipeline is automatically protected without the pipeline code needing to set retention headers.

### 4. S3 Glacier Vault Lock: Policy-Based Archival Compliance

Glacier Vault Lock is S3 Glacier's equivalent of Object Lock but applied at the vault/bucket level via a Vault Lock Policy. It is specifically designed for long-term archival in the 7-year+ compliance window.

**The locking process is two-phase.** First, you initiate the Vault Lock — this creates a policy with a 24-hour grace period during which you can test and validate it. Second, you complete the lock. Once completed, the Vault Lock Policy cannot be changed or removed. This two-phase pattern exists because a permanent Vault Lock on a restrictive policy could make data permanently inaccessible.

**What Vault Lock prevents.** A Vault Lock policy can deny any ability to delete archives before a retention period expires, deny any attempt to disable the vault, and enforce MFA-authenticated deletes. Even a root account user in the account cannot override a completed Vault Lock policy.

**When to use Vault Lock vs Object Lock.** Vault Lock is for S3 Glacier vaults (the legacy Glacier service separate from S3 Glacier storage classes). Object Lock is for data stored in S3 buckets using the Glacier storage class via S3 Lifecycle rules. For new workloads, use S3 Object Lock with the Glacier Flexible Retrieval storage class — this is the modern, integrated pattern. Vault Lock is primarily relevant for legacy Glacier vault workloads or when pure S3 Glacier API access is required.

### 5. MFA Delete: Adding Human Authorization to Permanent Deletes

MFA Delete is an additional protection layer on a versioned S3 bucket. When enabled, permanently deleting an object version or disabling versioning requires the root account to provide a valid MFA code in addition to valid AWS credentials.

**Who can enable/disable MFA Delete.** Only the bucket owner using the root account (not an IAM user or role) can enable or disable MFA Delete. This is intentional — it prevents an IAM user whose credentials are compromised from disabling MFA Delete and then mass-deleting object versions.

**Practical application.** MFA Delete is appropriate for the highest-sensitivity data stores: primary financial records, compliance archives, backup buckets. For most operational buckets, Object Lock in Governance mode is a better everyday protection mechanism. MFA Delete specifically protects against compromised IAM credentials being used to permanently delete data.

### Module 1: Encryption Models — Configure and Observe

**Goal:** Set up each encryption model and verify it works end to end with CloudTrail evidence.

#### Lab 1A: SSE-S3 Default Encryption

1. Create a bucket with no encryption specified. Go to Properties → Default encryption → SSE-S3.
2. Upload an object without specifying any encryption header.
3. Select the uploaded object → Properties → Server-side encryption method. Verify: SSE-S3.
4. Note: no KMS CloudTrail event was generated. Open CloudTrail → check for `kms:GenerateDataKey` events — none should appear.

#### Lab 1B: SSE-KMS with Bucket Key

1. Create (or use an existing) AWS KMS Customer Managed Key.
2. Set bucket default encryption to SSE-KMS, using your CMK ARN. Enable Bucket Key.
3. Upload an object.
4. Open CloudTrail → Event history → filter by Event Source `kms.amazonaws.com`. Look for `GenerateDataKey` events.
5. Upload 10 objects in rapid succession. With Bucket Key enabled, observe that KMS events are drastically fewer than 10, because the bucket-level data key is cached.

#### Lab 1C: Enforce SSE-KMS via Bucket Policy

1. Add the following bucket policy to deny any PUT that does not specify SSE-KMS:
   ```json
   {
     "Version": "2012-10-17",
     "Statement": [{
       "Effect": "Deny",
       "Principal": "*",
       "Action": "s3:PutObject",
       "Resource": "arn:aws:s3:::your-bucket/*",
       "Condition": {
         "StringNotEquals": {
           "s3:x-amz-server-side-encryption": "aws:kms"
         }
       }
     }]
   }
   ```
2. Try to upload an object without specifying encryption:
   ```bash
   aws s3 cp test.txt s3://your-bucket/no-encrypt.txt
   # Should fail with 403 AccessDenied
   ```
3. Upload with the correct header:
   ```bash
   aws s3 cp test.txt s3://your-bucket/kms-encrypt.txt \
     --sse aws:kms --sse-kms-key-id <your-key-id>
   # Should succeed
   ```

#### Lab 1D: Enforce HTTPS-Only (In-Transit Encryption)

1. Add the TLS-enforcement bucket policy:
   ```json
   {
     "Effect": "Deny",
     "Principal": "*",
     "Action": "s3:*",
     "Resource": ["arn:aws:s3:::your-bucket", "arn:aws:s3:::your-bucket/*"],
     "Condition": {"Bool": {"aws:SecureTransport": "false"}}
   }
   ```
2. Attempt to access the bucket over HTTP (most CLIs default to HTTPS, so this test is theoretical unless you configure an HTTP-only client). Verify the policy is in place via the bucket's Permissions tab.

### Module 2: S3 Object Lock — WORM Retention

**Goal:** Apply Object Lock to protect objects from deletion and understand the behavior difference between Compliance and Governance modes.

#### Lab 2A: Create an Object Lock Bucket

1. Create a new bucket. In the creation wizard, enable **Object Lock** (this also enables Versioning automatically).
2. **Important:** Object Lock cannot be enabled after bucket creation.
3. Set a default retention period: 1 day, Governance mode.

#### Lab 2B: Upload and Verify Locked Object

1. Upload a file to the bucket.
2. Select the object → Object Lock → View and modify Object Lock settings. Verify the retention expiry date shows tomorrow's date.
3. Try to delete the object by specifying its version ID:
   ```bash
   aws s3api delete-object \
     --bucket your-object-lock-bucket \
     --key test.txt \
     --version-id <version-id>
   ```
4. Observe the error: `AccessDenied`. The object is protected by the governance retention.

#### Lab 2C: Bypass Governance Mode (Authorized Deletion)

1. Add an IAM policy granting `s3:BypassGovernanceRetention` to your user/role.
2. Retry the delete with the bypass header:
   ```bash
   aws s3api delete-object \
     --bucket your-object-lock-bucket \
     --key test.txt \
     --version-id <version-id> \
     --bypass-governance-retention
   ```
3. This succeeds, confirming Governance mode allows bypass by authorized principals.

**What to note:** In Compliance mode, the same bypass-governance-retention flag is silently ignored. No IAM principal, including root, can delete or shorten the retention period of an object in Compliance mode before expiry.

#### Lab 2D: Legal Hold

1. Apply a legal hold to an object:
   ```bash
   aws s3api put-object-legal-hold \
     --bucket your-object-lock-bucket \
     --key test.txt \
     --version-id <version-id> \
     --legal-hold Status=ON
   ```
2. Attempt to delete even with bypass-governance-retention flag. Observe the legal hold prevents deletion regardless.
3. Remove the legal hold before the lab objects can be cleaned up:
   ```bash
   aws s3api put-object-legal-hold \
     --bucket your-object-lock-bucket \
     --key test.txt \
     --version-id <version-id> \
     --legal-hold Status=OFF
   ```

### Module 3: Azure Blob Immutability (Equivalent Pattern)

**Goal:** Configure time-based retention and legal holds on Azure Blob Storage.

1. Azure Portal → your Storage Account → Containers → Create a container.
2. Container → Access Policy → Immutable blob storage → Add policy.
3. Policy type: Time-based retention. Retention period: 1 day.
4. Save (do not Lock — the locked state is irreversible).
5. Upload a blob. Attempt to delete it — Azure denies the delete while the immutability policy is active.
6. Add a Legal Hold: same Access Policy view → Legal holds → Add identifier.
7. Observe that legal hold prevents deletion even if the retention period expires.

**Comparison with AWS Object Lock:**
- Azure: Policy applies at the container level; locks propagate to all blobs in the container.
- AWS: Object Lock applies at the individual object (version) level OR via bucket default retention.
- Both: locked state in Compliance/Locked mode is irreversible.

## Troubleshooting Guide

| Symptom | Most Likely Cause | Diagnostic Steps |
|---|---|---|
| `AccessDenied` on PutObject with expected SSE-KMS | Bucket policy denies PUT without encryption header; KMS key policy does not allow the calling IAM role to use the key | Check bucket policy for `StringNotEquals s3:x-amz-server-side-encryption`; check KMS key policy for `kms:GenerateDataKey` permission for the caller |
| SSE-KMS throughput degraded / KMS throttling | High request rate hitting KMS per-request limit (30,000 req/s burst, region-wide) | Enable S3 Bucket Key to cache the data encryption key at the bucket level; check KMS CloudWatch metrics for throttling |
| Object in Compliance mode cannot be deleted | Compliance mode: no deletion possible before retention expiry, regardless of IAM permissions | Wait for retention period to expire. This is working as designed. There is no override. |
| Glacier restore returns `RestoreAlreadyInProgress` | A restore request is already queued for this object | Wait for the restore to complete; check object properties for `x-amz-restore` status |
| Glacier object download fails after restore request | Restore takes time: Expedited = 1-5 min, Standard = 3-5 hr, Bulk = 5-12 hr. Download before restore completes fails | Check object restore status in the console or via `head-object` API call; wait for `ongoing-request=false` in the restore metadata |
| Object Lock cannot be enabled on existing bucket | Object Lock can only be enabled at bucket creation time | Create a new bucket with Object Lock enabled; use S3 Batch Operations to copy existing objects to the new bucket |
| MFA Delete cannot be configured via console | MFA Delete can only be enabled on a bucket via the CLI or SDK using the root account credentials; the console does not support it | Use the AWS CLI: `aws s3api put-bucket-versioning --bucket <name> --versioning-configuration Status=Enabled,MFADelete=Enabled --mfa "<serial> <code>"` using root credentials |
| Vault Lock completed with wrong policy | Vault Lock policy is permanent once locked; the 24-hour grace period is the only window for correction | Always test Vault Lock policies in the 24-hour pending state before completing the lock. Verify with test archives before locking. |

## Quick Revision Sheet

**S3 Server-Side Encryption Models**
| Model | Key Owner | CloudTrail Key Audit | Per-Request KMS Cost | Revoking Access |
|---|---|---|---|---|
| SSE-S3 | AWS | No | No | Not possible at key level |
| SSE-KMS (AWS managed key) | AWS (aws/s3) | Yes | Yes | Not possible to revoke by policy |
| SSE-KMS (CMK) | Customer | Yes | Yes | Yes — disable or change key policy |
| SSE-C | Customer | No (key never stored) | No | Key must be provided per-request |
| Client-Side | Customer | No | No | Full client control |

**Object Lock Modes**
- **Compliance mode:** No one can delete before expiry, including root. Cannot shorten retention period. Regulatory-grade.
- **Governance mode:** `s3:BypassGovernanceRetention` permission allows override. Protects from accidents.
- **Legal hold:** Indefinite, no expiry. Requires explicit removal. Independent of retention mode.
- Object Lock must be enabled at bucket creation. Versioning is automatically enabled.
- Default bucket retention period applies to objects without explicit per-object retention settings.

**Glacier Vault Lock**
- Two-phase commitment: initiate (24-hour grace period) → complete (irreversible).
- Permanently prevents changes to the vault lock policy after completion.
- Use for legacy S3 Glacier vault workloads. New workloads: use S3 Object Lock + S3 Glacier storage class.

**MFA Delete**
- Only root account can enable via CLI.
- Required for: permanently deleting object versions, disabling versioning.
- Best for highest-sensitivity compliance buckets.
- Governance mode Object Lock is a better everyday control for most workloads.

**Encryption Decision Flow**
1. No compliance requirements, non-sensitive data → SSE-S3 (simplest, free).
2. Compliance auditing required (who decrypted what) → SSE-KMS with CMK + Bucket Key + CloudTrail.
3. Keys must never leave your premises → SSE-C or client-side encryption.
4. In-transit: always enforce HTTPS via `aws:SecureTransport: false` bucket policy deny.

## Troubleshooting Notes

| Symptom | Likely Cause | Check First |
|---|---|---|
| Unexpected AccessDenied | Bucket policy, Block Public Access, KMS policy, or object ownership mismatch | Check the caller principal, bucket policy evaluation, and encryption key policy together |
| Objects restore too slowly or remain unavailable | Archive tier retrieval has not completed or lifecycle changed the object state | Inspect the current storage class, restore status, and lifecycle timers |
| Transfer job stalls or never starts | Agent reachability, endpoint routing, or IAM trust is incomplete | Validate network reachability on port 443, task execution role, and endpoint resolution |
| Throughput is lower than expected | The chosen storage service or class does not match the access pattern | Check request concurrency, filesystem mode, and the service-specific throughput model |
| Cost is higher than expected | Duplicate retention, retrieval fees, or wrong class selection | Review lifecycle, replication scope, archive restores, and request-heavy access paths |
| Behavior contradicts the data protection archival model | The intended control boundary was never translated into a concrete policy, route, identity path, or recovery signal | Re-check the mental model, then inspect the first control plane, data plane, and evidence source that should prove it |
