-: In this lecture we will talk about, StatefulSets in Kubernetes. Now, in the previous lecture we discussed why you need a StatefulSet. Note that you might not always need a StatefulSet. It really depends on the kind of application you're trying to deploy. If the instances need to come up in a particular order, if the instances need a stable name, et cetera. So once you evaluate your requirements and decide that StatefulSet is the right choice for you, you can create a StatefulSet, just like how you created a deployment. Create a deployment definition file, with a pod definition template inside it. And then all you need to do, is change the kind to StatefulSet instead of deployment. Note that both SS in the StatefulSet is uppercase. Apart from that, a StatefulSet also requires a service name specified. You must specify the name of a headless service, we will discuss about headless services and why you need to specify a service name here in the upcoming lecture. As discussed before, when you create a StatefulSet using this definition file, it creates pods one after the other, that's ordered graceful deployment. Now each pod gets a stable, unique DNS record on the network that any other application can use to access a pod. When you scale the StatefulSet, it scales in an ordered graceful fashion. Where each pod comes up, becomes ready and only then the next one comes up. This helps when you want to scale MySQL databases, as each new instance can clone from the previous instance. It works in the reverse order when you scale it down. The last instance is removed first, followed by the second last one. The same is true on termination. When you delete a StatefulSet, the pods are deleted in the reverse order. Now, that's the default behavior of a StatefulSet, but you can override that behavior to cause StatefulSet to not follow an ordered launch. But still have the other benefits of StatefulSet such as a stable and unique network ID. For that, you could set the pod management policy field to parallel to instruct StatefulSet to not follow an ordered approach. Instead deploy all parts in parallel, the default value of this field is ordered ready.
