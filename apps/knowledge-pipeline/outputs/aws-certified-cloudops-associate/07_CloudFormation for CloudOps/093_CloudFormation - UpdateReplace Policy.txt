So now let's talk about UpdateReplacePolicy. So it controls what happens to a resource if you update a property whose update behavior is replacement. For example, if you change an RDS DatabaseInstance's AvailabilityZone property, then it's going to be replaced because an RDS database belongs to a specific AZ. And so by default, when you replace a resource in CloudFormation, it's going to delete the old one and create a new one. So it will delete the old resource and have a new one with a new physical ID. But if you have UpdateReplacePolicy=Retain, then the resource is going to be kept, and the new one is going to be created, but the old resource will still be removed from CloudFormation's scope. And if you have UpdateReplacePolicy=Snapshot, then you're going to have a snapshot of the old resource and the new resource being created. So this can apply, for example, to as well EBS volumes, ElastiCache Clusters, ElastiCache ReplicationGroup, and so on. RDS DatabaseInstance, RDS DatabaseCluster, Redshift Cluster, Neptune DBCluster. And the snapshot when made doesn't exist in CloudFormation scopes either, but it's nice for you in order not to lose data in case you did a wrong update. Now, what's the difference between UpdateReplacePolicy and DeletionPolicy? Well, UpdateReplacePolicy only applies to the resources replaced during stack updates, for example, for your databases. And DeletionPolicy applies to the resources deleted when a stack is deleted. And it applies when a resource definition is just gone.
