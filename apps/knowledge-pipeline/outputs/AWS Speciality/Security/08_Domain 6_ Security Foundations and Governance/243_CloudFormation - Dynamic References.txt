So now let's talk about CloudFormation Dynamic References. So the idea is that from within a CloudFormation template, you wanna reference external values that are stored in the SSM Parameter Store or Secrets Manager. And what's going to happen is that CloudFormation will retrieve the value during any stack, create and update operations. So we can store, for example, database credentials, password, third-party API keys, etc. in Parameter Store or Secrets Manager and they will be retrieved at runtime. So here's the example. CloudFormation is going to receive a template for create or update, and then it can get values from SSM or Secrets Manager. So we have three kind of dynamic references. We have the SSM Dynamic reference, and it's for plain text values stored in the SSM Parameter Store. Then we have the SSM Secure, which is for secure strings stored in SSM Parameter Store. So please notice here that SSM and SSM Secure indicate, one, plain text, the other one, Secure Strings. You cannot mix them together. And finally, we have Secrets Manager to retrieve secret value directly stored in Secrets Manager. And so the way it works is that in your CloudFormation template, you should have a resolve, then service name, and then reference key. So I'll show you in a second how that works. So say for example, we have this IAM user and we resolve an IAM password user from SSM, and we use SSM Secure. So it's going to be a secure value, so it's going to be encrypted. So in that case, confirmation will at runtime fetch the SSM Secure value from the SSM Parameter Store and then we'll be able to create an IAM user. For RDS, for example, here we use Secrets Manager, and again, the values are stored in Secrets Manager. So we have Resolve Secrets Manager, and then we fetch the secret value to create an RDS database instance. So hopefully you get the idea behind dynamic references and its benefits for security.
