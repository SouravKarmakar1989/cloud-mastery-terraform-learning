Well, for example, you can create a rule saying that every S3 bucket should be encrypted if created via CloudFormation. So for this, how do we do it? Well, we create these policies thanks to Domain-Specific Language, which looks like this. For example, we have a file named rules.guard. And in it we say that for the S3 bucket, the property of ServiceSideEncryptionConfiguration should say AES256. Otherwise, we're saying that it will fail because the bucket encryption is not configured with the correct algorithm. And so once we have this, we have our template.yaml file and we run the CLI tool to validate the YAML file against the rules that we have defined as part of CloudFormation Guard. And if it passes, we're good to go. If it doesn't pass, then we can start, for example, a CI/CD pipeline. So cfn-guard is not used to verify the syntax of your CloudFormation templates. It's used to verify if your CloudFormation templates obey some rules that you have defined as part of CloudFormation Guard. Okay, so that's it for this lecture.
