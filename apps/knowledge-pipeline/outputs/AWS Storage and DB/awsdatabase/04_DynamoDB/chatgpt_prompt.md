### Lecture: Understanding DynamoDB Capacity - Properly Size DynamoDB and Reduce Costs

So we're going to talk about a really important topic here when it comes to designing your DynamoDB table. This is how you can control how much data your table can handle, and how fast it can read, and how fast it can write. So when you create a DynamoDB table, you need to decide how you want AWS to manage your capacity. And there are two modes available. There's provisioned mode and on demand mode. So let's start with provisioned mode. And this is the one that's going to give you the most control. With provisioned mode you set the capacity. You decide how much read and write throughput your table has. And this is going to be measured using read capacity units and write capacity units. And so you'll basically say hey AWS I want my table to handle five reads and and ten writes per second or whatever. And DynamoDB will then reserve that capacity for you, and you'll pay for it by the hour, whether you use it all or not. You can also enable auto scaling, which is going to adjust the read and write capacity numbers for you. It'll adjust them up or down automatically when your workload changes. It's kind of like cruise control for your tables performance. Whatever your needs are, auto scaling is going to make sure that you have the right number of capacity units to handle those. So with provisioned capacity, you are planning your capacity ahead of time. You're paying for what you provision, but you can also enable auto scaling. Now let's compare that to on demand mode and on demand is really all about convenience. You know, here you're not specifying the number of read or write capacity units. DynamoDB will automatically adjust your capacity based on traffic. So it'll scale up and scale down in real time, and you'll pay for only what you use. The downside of this approach is it's going to be more expensive. It's going to cost roughly 2.5 times more than provisioned capacity. And this is the way that AWS handles a lot of their services. Local secondary indexes or LSI and global secondary Indexes or GSIs. A local secondary index gives you an alternative sort key while keeping the partition key the same as your base table. So maybe what I want to do here is create a local secondary index where rating is going to be the sort key for that secondary index, so I can have an index sorted by ratings with all of this information.
