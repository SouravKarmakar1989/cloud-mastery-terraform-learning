### Lecture: Introduction to Aurora

In this video, I'll introduce Aurora and we'll spend some time learning about the overall architecture and how Aurora databases work. So Aurora is an AWS only relational database. These Aurora database instances run only in AWS, but they support the open source interfaces of MySQL and Postgres. And so MySQL code can run just fine on Aurora. You can also choose Postgres instead of MySQL. Both of these are supported on Aurora databases and it is heavily optimized. So MySQL and Postgres are open source database solutions. Aurora is going to give us about three times the performance of Postgres and five times the performance of MySQL. And so it's heavily optimized to make it perform like a commercial database solution. We also have automatic failover protection. With Aurora we can failover to a secondary replicated instance in under a minute in most cases. And this failover automatically occurs. Aurora supports up to 15 read replicas that support auto scaling, so we can automatically add and remove read replicas in response to changes in demand. There's also auto scaling storage, so when data is written to the Aurora instance, if we're starting to run out of storage capacity, additional storage capacity can automatically be added. And then we also have support for an Aurora global database, which can be spread across multiple regions. So let's start by taking a look at the basic architecture here. Here we have our primary database instance. And a lot of the concepts here are going to be very similar to RDS. So in this case we've got an Aurora database instance. And this is the primary instance. It supports reads and writes. And every aurora cluster has one instance that's considered a primary instance. And we can also have some replicas. So we can have up to 15 read replicas. These support read operations, but just like RDS, all of the write operations need to be performed on the primary. And we can have replicas in different availability zones. Now these are very different than read replicas in RDS. And the biggest difference is these replicas provide high availability. Aurora can automatically fail over to one of these replicas if the primary database instance fails. That wasn't possible with read replicas in RDS. And you can even specify the failover priority for different replicas. So if I have multiple replicas, I can choose which one is going to take over if the primary database instance fails. For storage, we have something called a cluster volume. This is SSD storage, and even if we're running a primary instance with no replicas, this cluster volume is still used. So the contents of the database are spread across storage systems in three availability zones, and the primary can read and write to this cluster volume. Any changes to the database are written to this cluster volume across all three availability zones. So now we have a copy of all of our database contents in multiple availability zones, even if we're not using replicas. But in this case we are using replicas. And the replicas are providing scaling, but they're also providing high availability. And the replicas will read from the copies of the data in their own availability zones. The database instances within the cluster must be in the three availability zones that the cluster volume resides in. So let's imagine that we have a primary instance and we have a single Aurora replica. And because of the cluster volume that we're using, there's already a copy of all of the required data in the three availability zones shown here. So when a new replica is created, there is no need to create a snapshot. There's no need to copy all of that data with RDS. If we wanted to create a read replica, we had to wait for all of the data to be copied. We had to wait for a snapshot to be created. But with Aurora, the storage is independent of the instances. So we've got this cluster volume. If I create a new replica in our cluster, it will simply attach to the cluster volume. So there's a big difference between the architecture of Aurora and RDS. And just to reiterate, some of the most important pieces of that with Aurora, just like RDS, we're going to have a primary database. We're going to have one database instance that serves as the primary that is performing all of the write operations. We can also have replicas, but unlike RDS, we can fail over to these replicas. Also, unlike RDS, even if we only have a primary database instance, we're still going to have this cluster volume that is spread across three availability zones. Any changes are written to all three availability zones, and that way when I add a new replica, it can just access the existing storage within its own availability zone. And we don't have to create a snapshot in order to spin up a new replica.

### Lecture: Aurora Endpoints

In this video, we'll dive deeper into the architecture of Aurora and learn about the different end points that it uses. And we're going to start with the cluster or writer end point. So with Aurora replicas you can have multiple instances. You can have the primary and the replicas. All of the application writes take place on the primary, but reads can be pointed to any of the replicas. So the cluster end point always represents the primary instance of the cluster. And think of this kind of similar to the DNS name that we learned about with RDS. So the database client doesn't know the IP address of the database server. It knows it by the cluster endpoint. And if that primary instance fails, the cluster endpoint can be updated to point all of the database activity to one of the replicas, which is now going to take over as the new primary. This all happens automatically. You don't have to manually fail over. So let's kind of ignore the primary instance for a moment and remove it from this diagram. Reads can be pointed to any of the replicas, and we have this end point called a reader end point. The reader end point provides load balancing for read only connections to the replicas. This eliminates all of the read workloads on the primary instance. And if the cluster contains only a primary instance and it doesn't have any replicas, the reader endpoint would connect to the primary instance in that case. So the reader endpoint is going to load balance connections, not the individual queries, but but the connections that are performing reads. And so because of this with RDDs, we had to take our database client and point it to a specific read replica. So if I had four replicas, I would have to individually configure my database clients, and maybe one client would point to this replica and another client would point to this replica. And with RDDs, I could have multi-az on my read replicas. So I would have to enable Multi-az on this read replica and Multi-az on this read replica in order for it to be highly available. This is a huge difference with Aurora, because the reader endpoint is going to receive all of these database client connections that are going to be doing reads, and it's going to distribute those connections across all of these read replicas. So now if one of these read replicas fails, it's no big deal. The database client still continues to use the same reader endpoint, and the reader endpoint will just send those connections to the replicas that are still working. We can also have custom endpoints. Custom endpoints are intended for advanced users with specialized kinds of workloads, where it isn't practical to keep all of the replicas the same. So here we see an Aurora cluster, but we've got different instance types mixed together in the same cluster. And in that case, I could create a custom endpoint that's going to load balance traffic across certain instances within my cluster. So in this case I've created a custom endpoint that's sending traffic to only the instances in the cluster that are the R7G extra large instance type. So in this scenario, maybe production traffic with higher priority gets load balanced across these more powerful instances. And then I can have another custom endpoint for my lower priority workloads that is distributing traffic across the less powerful database instances.

### Lecture: Aurora Storage and Caching

In this video, we'll learn about aurora storage and caching. Now we've got a couple of different options here for storage, and it's sort of similar to concepts that you might be familiar with for S3. So if you're comfortable with S3, you understand that S3 has S3 standard and S3 infrequent access. This is going to be very similar to that. There's going to be a storage class for Aurora called IO optimized, and IO optimized is going to be more expensive, but it's going to give better performance. IO optimized is ideal for IO intensive databases. This is the best choice when your IO spending is 25% or more of your total Aurora database spending. And with IO optimized, you're not paying for reads and writes versus Aurora standard. This is a more cost effective option for databases that have moderate IO usage. You're going to pay per million IO operations. And so the first option is going to give you unlimited IO operations. But it's going to be more expensive per gigabyte. Aurora standard is going to be less expensive per gigabyte. But you're going to pay for all of the I o operations that hit it, and you can always switch from one to another. So if you find that, hey, I've used Aurora Standard Storage, but I need Aurora IO optimized. You can switch once per 30 days so you're not locked into a specific storage type. So let's take a moment to talk about caching now. And I'm going to start with a bit of an analogy here. Let's say that we have a math problem that we want to solve. And if I need to solve this problem I could use a calculator. And it'll take a little bit of time and work on my part, and I'll come up with a solution for it. And let's say that I take the results of this problem and I write it down on a piece of note paper. That's kind of like storing the results in a cache. If somebody else asks me that exact same math question, I don't need to break out the calculator. I don't need to do any math. I've already got the results stored here in my cache and I can provide them instantaneously. So the benefits of this are number one, I can provide a result faster. And number two, every time that problem is presented to me, I have less manual effort because I've already figured out the result. That's kind of like caching for a database. And Aurora automatically has a built in right through cache. So whenever data is written to my Aurora database, the cache is updated. Now there's only a certain amount of space in this memory cache, and the amount of space is limited to the memory provided by the instance itself. Different instance types come with more memory, and the cache is going to consume some of that memory. But now let's imagine that a query is created and sent to my database. This is going to consume resources on the database. And therefore the database needs to be able to handle all of those queries that are coming in. But if I have a cache that is storing many of those query results, let's now imagine that that same query comes in again. Well, now the database doesn't have to figure out the results of that query. It's already done it, and it's already stored the results of that query in its cache. So therefore future queries that are identical all won't require the resources of the instance. They will be stored in memory, and they'll be provided much more quickly.

### Lecture: Aurora Network Security and Authentication

In this video, we'll learn about some of the concepts of Aurora security. And specifically we're going to focus on network security and authentication. And so aurora clusters are always launched within a VPC. So here we see our VPC. And just like RDS the aurora cluster is launched inside of a VPC. And they should be placed in a private subnet in most cases. So the private subnet is going to have no direct internet access. And in this case the database instance is not going to get a public IP. And I might have things like EC2 instances and application servers or web servers that are internet facing. So the EC2 instance, maybe this is my web server that can be in a public subnet that could potentially be accessible from the internet, but the database itself sits in a private subnet, and that database is never going to be directly accessible from the internet. And the way that I can do that is similar to RDS. I can use a security group. The security group can specify the source of traffic. So for example, the source in this case is this EC2 instance. The security group is only going to allow database traffic on the correct port from this EC2 instance. Or the EC2 instance itself could be in a security group, and that could be the source that is allowed. So I could allow all traffic from that specific security group on a specific port. All right. So now let's take a moment to talk about Aurora IAM authentication. So you could take the same approach as with a standalone database instance. You could just set up database authentication with username and password. And that's pretty straightforward. Or we could create an IAM role with the necessary privileges and assign that role to an EC2 instance. The role is going to be used by the EC2 instance to get an authentication token. And we will then create a database user account that uses that authentication token. This is very similar to what we saw with RDS. So in this situation we've actually assigned a specific role to the EC2 instance. And the role is essentially allowing the EC2 instance to get this particular token. This is how the EC2 instance is authenticating to the database. Once it has that token, there is a user associated with the database that allows the EC2 instance to now have access to that database. And the big benefit of this is I do not need to embed database credentials into the EC2 instance. This is important for a few reasons. Number one, I don't want to store my database credentials inside the operating system of an instance because that's a security risk. If somebody were to compromise this instance, they could potentially get access to those security credentials for the database. And the other benefit of this approach is I don't have to manage usernames and passwords inside of my EC2 instance, so I may want to change my database credentials every so often. If I have many database clients that all have those credentials stored. Now that's a manual operation to go into all of those instances and change those credentials. So I am authentication can allow us to avoid that and just allow the EC2 instance to assume a role and get access to the database based upon that role.
