Welcome back. This is going to be a really quick lesson on the legacy terminology used to create or refer to Delta live tables. As with the rest of Databricks, Delta Live Tables functionality has evolved significantly over the last couple of years. So did the terminology. Please note that Databricks still supports all terminology for backward compatibility, and you may come across them whilst working in the industry, projects, documentation, or even in the certification exam. So please be aware of them. Firstly, materialized views used to be called live tables, and the syntax for creating a materialized view was a create or refresh live table instead of create or refresh materialized view. Now, if you run either of those statements, that will create a materialized view. Similarly, streaming tables used to be called streaming live tables. Again, the syntax for creating the streaming table or streaming live table was create or refresh streaming live table instead of create or refresh streaming table. Again, both statements today means the same thing and they simply create a streaming table. Databricks fully supports the legacy syntax for backward compatibility, and I don't see that going away anytime soon. So if you see live table or streaming live table in the documentation, they simply refer to Materialized View and Streaming table respectively. Please be aware of this. I still see the legacy terminology used in production projects white papers, medium articles, LinkedIn articles, etc.. Databricks recommends using the new terminology and the syntax going forward, but please be aware of the old terminology as well as the syntax. So if you come across them, you know what they mean.
