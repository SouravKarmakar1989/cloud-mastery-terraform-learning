So now let's talk about Amazon S3 object integrity. So the idea is that you upload objects into Amazon S3 and you want to make sure that the objects you upload are the ones you had beforehand. So for this we'll use the MD5 checksum. So the idea is that you have a file, it's made out of bytes and the checksum is an algorithm that will be running, the MD5 is a specific kind of algorithm, and it will give you a string of a certain length and that string is supposed to be unique for that file. So the idea is that if you upload a file and you compute an MD5 on your own machine and then Amazon S3 also computes the MD5 on their own servers, in case they match then the objects we have uploaded is the same as we had on our servers or on computers. And if the MD5 checksums are different then the upload was not successful. So to make sure that Amazon S3 accepts or rejects a upload we can pass in a header named Content MD5 as part of our HDP request. If we want to do it the other way, for example, we want to compare an object on Amazon S3 and make sure we have downloaded it correctly or that we have a copy of it that is the same as the one online, then Amazon S3 has an ETag on your object and the ETag is equal to the MD5 hash or checksum if you're using SSC S3. And so the idea is that we use the API call get object metadata so that we only download the ETag, not the object itself. And the ETag is going to be compared to our local version of the object where we can also compute the MD5. And again, if the two are matching then we know we have the same object as the one on Amazon S3. But if they're not matching we know we have a different object. So other checksums are supported other than MD5. You have SHA-1, SHA-256, CRC32, and CRC32C. So that's it for this lecture. I hope you liked it and I will see you in the next lecture.
