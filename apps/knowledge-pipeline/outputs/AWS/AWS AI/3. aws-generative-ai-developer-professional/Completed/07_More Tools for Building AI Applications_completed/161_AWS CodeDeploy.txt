So now let's talk about AWS CodeDeploy. So as the name indicates, it is a deployment service that will automate application deployment. What does that mean? Well, that means you have a version one of an application running and somehow, you want to upgrade it to version two. Now where can we deploy version one and version two? Well, these application versions can be deployed either to EC2 instances, on-premises servers or Lambda functions or ECS services so three different kinds of targets because EC2 instances and on-premises servers have the same methodology. On top of it, CodeDeploy can allow you to update an application but also rollback. So in case your deployment is not working as expected, it is failing, or there's an alarm that gets triggered, then we can automatically initiate a rollback. That makes it a very safe automated deployment. On top of it, we can control how fast we want to deploy our application. We will see this later on, but we can say one instance at a time or all at a time or HalfAtATime or blue/green. Finally, to control how CodeDeploy works, there's a file named appspec.yml that can help define how the deployment will happen. So now, let's talk about the EC2/on-premises platform. So this allows you to deploy to EC2 instances and on-premises servers, your application, and your code. And you can perform two kind of deployments, either in-place deployments or blue/green deployments. We'll see them both to understand the difference. And to work, we must first deploy the CodeDeploy agent onto the target instances because the agent is the thing that will be performing the updates on the instances. We can define the deployment speed so, as I said, we can have a gradual deployment. For example, we can do AllAtOnce, which will update all the instances at once, but will give us the most amount of downtime because we will be down. HalfAtATime, for example, to reduce capacity by 50% or OneAtATime, which is going to be the slowest because we take down one instance at a time and update it, but of course, it's going to give us the lowest availability impact because only one instance will be down at a time or you can define your own custom deployment speed if you wanted to. So let's have a look at how, for example, in-place deployment works for HalfAtATime setting. So here we have the v1 running on four EC2 instances. Because we're in a HalfAtATime setting in in-place deployment, two instances are going to be taken down for maintenance. So the agent will stop the application on two instances which then are going to be upgraded again by the agent to version two. And once it is done, then the other half is going to be taken down for install and then upgrade it to version two. And so this is the process of in-place deployment for HalfAtATime setting. What about blue/green? Well, blue/green is the idea that you have, for example, an Application Load Balancer pointing to version one and all your instances are in Auto Scaling group. What's going to happen is that we're going to create a new Auto Scaling group so this can be done either manually or automatically by CodeDeploy. And CodeDeploy is going to create as many EC2 instances in this new Auto Scaling group. So we have v1 and v2 now running in parallel and the load balancer through some process is going to be pointing now to the v2 Auto Scaling group and the v1 will be taken down. So this is a simplified version, but at least you see the process, and it's called blue/green because new instances are taken up and the old ones are terminated. So this CodeDeploy agent I've been telling you about, it must be installed on the EC2 instances as a prerequisite. So we have the EC2 instance, and we install the CodeDeploy agent. Now you can install it directly using some Linux commands to install any software or if you wanted to have some kind of automation. If your EC2 instances are managed using Systems Manager, then you can automate the install of CodeDeploy agent using Systems Manager. Now this EC2 instance with the CodeDeploy agent will need sufficient IAM permissions to access Amazon S3. Why? Well, because in Amazon S3 will be stored the application revision. So when we want to update from version one to version two, then the CodeDeploy agent with the proper IAM permissions attached to the EC2 instance will download the application revision from the S3 bucket. And so that allows us to just use the SAM framework and internally use CodeDeploy. So how does that work for the traffic-shifting function? Well, say we have a PROD alias, okay? And we have v1 that we've deployed before and v2 that we've deployed after, and we wanna shift the traffic between the v1 version and the v2 version under the PROD alias. So the way it works is that CodeDeploy is going to make this X vary until X equals 100%. So at first, X will be zero. So that means that all the PROD alias points to the v1 function, and then X will gradually increase until he reaches 100% so that all the traffic on the PROD alias is redirecting to the v2 function. Now we have several strategies to make X evolve. So we have linear where we grow the traffic every N minutes until 100%. So in this example, LambdaLinear10PercentEvery3Minutes, that means that every three minutes, 10% will be added. So we'll have 10 and then 20 and then 30% after 9 minutes, and so on until we reach 100%. And so you have different strategies. For example, 10PercentEvery10Minutes and so on. So this is linear. Then we have canary, which is to have a small amount of traffic on the v2 function and then ramp up to 100%. So here we have LambdaCanary10Percent5Minutes. So that means that for five minutes, the new v2 function is going to receive 10% of the traffic. And then, if everything goes well, then all of a sudden, we switch to 100% of the traffic. That's why it's called canary. So we have, for example, the 10Percent5Minutes or the 10Percent30Minutes strategy. And finally, AllAtOnce, which immediately jumps from v1 to v2 but we don't have the time to test it if it's working. Now, CodeDeploy for the ECS platform is actually similar.
