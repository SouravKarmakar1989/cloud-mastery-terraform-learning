### Lecture: AWS Glue

All right. Let's talk about AWS glue, which is a system for building your table definitions and performing ETL, extract, transform and loading of your data automatically. Now glue is a huge deal on the exam right now. So back when they removed all of the machine learning content from the AWS Big Data Exam, it was called at the time into its own exam for machine learning. They kind of filled that gap mostly with more content about glue and redshift. So glue really plays a major role in the exam today. So it's very important to pay attention to this section. There's not really a whole lot you need to know about glue, but you'd need to really internalize it and understand how it connects all these different services together and how it might be used as pieces of larger applications. So what is glue all about? Well, it's a serverless system. So again, you know, there's nothing you need to maintain there that will automatically handle discovery and definition of table definitions and schema. So its main use is to serve as a central metadata repository for your data lake. It will discover those schemas out of your unstructured data, sitting in S3 or whatever, and publish table definitions for use with analysis tools such as Athena or Redshift or EMR. So again, the purpose of glue itself is to extract structure from your unstructured data. If you have data sitting in a data lake in S3, it can provide a schema for that so that you can query it using SQL or SQL like tools, including RDS and Redshift and Athena and Amazon EMR, and pretty much any other SQL database that can use a schema like that. The other thing glue does is custom ETL jobs. So once it discovers that schema, it can do some processing of it for you. And this can be trigger driven based on when that data is received or on a schedule or on demand. And it's fully managed. So again, there's no services for you to maintain, no hardware for you to maintain the ETL jobs. Use Apache Spark under the hood. We haven't talked about that yet, but spark is a very popular system for distributed data processing. But you don't need to manage the spark cluster, which is really nice. So you have the entire power of Apache Spark behind you with glue ETL without having to manage that spark cluster, which is a really, really nice thing. So one piece of it is the glue crawler and data catalog, which it populates. So the purpose of the Amazon glue crawler is to scan data in S3. And often it will infer that schema automatically just based on the data that it sees sitting there. But sometimes you have to give it some hints. You can schedule that to run periodically if you need to. And what that does is populate the glue data catalog. So we have data sitting in S3. Glue can periodically or on demand scan that data and construct a schema of it. Again that's just the table definition. You know, what are the columns of data in that information that you have stored there in an unstructured manner? What's their data types? Things like that. Column names, it can infer that as well, where it's stored. That's all important information as well. And it uses that information to allow things like redshift, Athena or systems running on EMR like hive to query your unstructured data in S3 as if it were structured data just like it were a traditional relational database. Now the data itself remains where it was originally, so the data stays in S3. You're not duplicating that data. We're not copying it into an actual database table somewhere else. We're just using glue to basically provide the glue between these relational database interfaces and the unstructured data lake that's sitting back here in S3. Once cataloged, that data is immediately available for analysis, and we'll build this out and go into a lot more detail into these systems later on. And then, you know, once you actually have that data sitting in something like Athena, you can use something like Amazon QuickSight to visualize that data. So glue again is the glue between your unstructured data and structured data analytics tools. So let's talk about Glu and S3 partitions really quickly here. So it's not entirely magic. You know, you do need to think about how you're going to structure your unstructured data. Now, the glue crawler will extract partitions of your data based on how your S3 data is organized. So you want to think upfront about how you're going to be querying your data lake in S3, and what those partitions should represent for the best performance. On the other hand, if you're querying primarily by device and not by time, you might want to put the device up first. That way, your primary partition would be the device ID and then followed by the year, month, and date.
