Now let's talk about Termination Policies in your Auto Scanning Group. So this allows you to decide which instances get terminated in case of a scale-in event or an Instance Refresh or an AZ Rebalancing, that means when you have too many instances in one Availability Zone and they get terminated to rebalance it. So there is the Default Termination Policy, which is quite simple to explain. You select the Availability Zone with the more instances and then you terminate the instance with the oldest Launch Template or Launch Configuration. And then if they were launched with the same Launch Template, then the instance that gets terminated is the one that is the closest to its billing hour. So let's take an example. We have the Auto Scaling Group with two EC2 instances started with an old Launch Template and then the new Launch Template allows us to keep on scaling out and having now five EC2 instances. Now, there is a scale-in event because we don't need as much capacity. So we first evaluate the AZ with the most amount of instances, so that's going to be AZ-A. And then we look at the Launch Templates and we choose the one with the oldest Launch Templates, so the first one, and we terminate it. If we had two with the same old Launch Templates, we would choose the one with the closest next billing hour, and that's it. Now you can customize these Termination Policies to either have the Default one, which is the one I just explained, or to have different ones. For example, this one is called AllocationStrategy. And so it will terminate instances to remain in line with your Allocation Strategy. So for example, if you're using Spot Instances and you choose lowest-price, it will choose to remain in line with the lowest-price allocation for your Spot Instances. Or if you're using On-Demand Instances with a wide variety of instances, then you would choose the one with the lower priority. Also, you have OldestLaunchTemplate to terminate instances that have the oldest Launch Template. Or OldestLaunchConfiguration for the oldest Launch Configuration. ClosestToNextInstanceHour to terminate the instances that are going to be closest to the next billing hour. NewestInstance to actually terminate the newest instance, which is very helpful if you're testing a new Launch Template, but then you're scaling back in and it will automatically terminate these new instances you've created. OldestLaunchTemplate, very helpful to terminate your old instances. So this is very helpful and this is different from Launch Configuration because, or Launch Templates, because if you're upgrading the instance sites, for example, you go from m5.large to m5.4xlarge, then you're not changing the Launch Templates, but you're changing the instance size. In this case, you want to delete your oldest instances. And you can choose one or more policies and actually specify the evaluation order. So you can say, you can combine OldestInstance with ClosestToNextInstanceHour, and then AllocationStrategy. Whatever you want.
