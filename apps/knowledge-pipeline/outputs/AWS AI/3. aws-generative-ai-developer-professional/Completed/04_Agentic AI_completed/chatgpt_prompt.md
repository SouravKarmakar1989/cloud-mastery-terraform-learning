### Lecture: Multi-Agent Workflows

All right. So what we've seen so far are single agent systems, where we have one foundation model that's been armed with a suite of tools that it can use to do stuff with, like looking up information, doing stuff on your computer, hitting the internet, whatever it might be. That's what we call a single agent system, or sometimes an augmented Lem. But what if we put many of these things together? That opens up even more possibilities. And that is the world of multi-agent systems. So let's dive into an example and dig into why we might want to do that. So you can kind of think of a multi-agent system as similar to giving tools to a single agent. But the difference is that we have many tools being used at once, potentially. So these agents can all be specialized for different tasks, and they can run independently. They can run in sequence, they can run in parallel. And even more importantly, they're all self-contained. So they can just focus on their task at hand without dealing with the context of all this other stuff in your system. And that's a powerful thing. An example is coding agents. So, um, I might have a coding agent that wants to operate on multiple files in different ways to achieve a larger task. You know, maybe I have, you know, a coding agent that's focused on code reviews, another one that's focused on optimization, another one that's focused on some sort of re-architecture. Right. Those could all be operating on the same code base, maybe even in sequence. Right. Another example would be a workflow that's requiring some sort of complex decision making, and that can benefit from more complex agent workflows. Now, I do want to warn you, if you do have a simple use case, you should not go with a complex solution like a multi-agent system. So only use these when you really need to when you are given a really complicated problem. In that case, multi-agent systems can help. This particular pattern is called the orchestrator pattern. And let's start with that one. I'm going to talk about this more in a subsequent slide. So I'm going to kind of gloss over it here. But this is just an example to kind of wrap your head around here. So in this example we have an orchestrator agent whose only job is to figure out how to break the task at hand in the prompt into multiple parallel, uh, pieces of work that need to be done. And then those are farmed out to individual specialized agents that are tuned and have the tools needed to achieve those specific tasks, and then a synthesizer pulls it all together at the end to actually generate your final response. So why would I want to use multiple agents or multiple augmented LMS, depending on your terminology? Well, one use case would be if you have too many tools. So if you find yourself with a single foundation model with a huge suite of tools, it might have trouble choosing the right one. And even more of a problem, it might have too much of a problem focusing on what it's doing because of the limited context space. So if you have an agent who's having trouble choosing the appropriate tool for the job, just like with a human being, that's not going to result in optimal results. So in that case, you might be better off having multiple specialized agents with a specialized set of tools for what they specialize in doing. Right. So, you know, it's kind of well, having specialists, you know, just like in human society. Another use case would be if you have very complicated logic.
